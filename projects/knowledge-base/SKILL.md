# 经营分析知识库 — 操作手册

知识库路径：`~/Claude/projects/knowledge-base/`
Schema 规范：`_schema.md`
操作日志：`wiki/log.md`
全库导航：`wiki/index.md`

---

## 操作一：Ingest（新文档入库）

**触发方式：** 用户提供新文档，说"Ingest 这份文档"或"把这个存入知识库"

**执行步骤：**

1. **保存原文**
   - 将原始文档存入 `sources/` 对应子目录，文件名加 `-raw` 后缀
   - 原文一旦存入，不再修改

2. **分析文档**
   - 读取原文，提取：核心结论、关键数据点、决策或 Action、异常现象、涉及的指标
   - 判断本文档涉及哪些 wiki 页面（可能同时影响品类页、决策页、指标页）

3. **更新 wiki 页面**（每份文档通常更新 3-8 个页面）
   - 若页面已存在：在对应章节追加新信息，标注来源文档和时间；如有过期内容，修正或标注"已更新"
   - 若页面不存在：按 `_schema.md` 中的模板新建，同步更新 `wiki/index.md`
   - 每个 wiki 页面更新后，刷新 `last_updated` 字段和 `summary` 字段

4. **更新 index.md**
   - 如有新建页面，在 `wiki/index.md` 对应分类下追加一行

5. **写入操作日志**
   - 在 `wiki/log.md` 末尾追加：
     ```
     ## {YYYY-MM-DD HH:MM} — Ingest
     - 操作对象：{文档名}
     - 更新页面：{页面1}、{页面2}、...
     - 备注：{简要说明新增了什么关键信息}
     ```

**注意：**
- 每条写入 wiki 的信息必须能溯源到 sources/ 里的原文
- 同一品类的信息追加在已有页面，不新建重复页面
- 不写推测性内容，只写文档中明确记载的事实

## 操作二：Query（查询）

**触发方式：** 用户提问，说"查一下..."或"知识库里有没有关于..."

**执行步骤：**

1. **定位相关页面**
   - 读 `wiki/index.md`，判断问题涉及哪些页面（品类/指标/决策/情报）
   - 优先读 summary，决定是否需要读全文

2. **读取相关 wiki 页面**
   - 读取命中的页面全文
   - 如页面有交叉链接，按需追踪读取

3. **综合回答**
   - 基于 wiki 内容回答，引用具体页面和时间

4. **File Back（可选）**
   - 如果回答过程中产生了新的综合洞察（非原文直接摘录，而是跨页面推断），
     将其归档回对应 wiki 页面，并写一条 log 条目
   - 格式：在页面相关章节追加，注明"综合洞察 · {日期}"

## 操作三：Lint（健康检查）

**触发方式：** 用户说"做一次 Lint"或"检查知识库健康状态"，建议每月触发一次

**执行步骤：**

1. **读取全量 wiki 页面**（通过 index.md 导航）

2. **检查以下问题并列出**：

   | 问题类型 | 检查方式 |
   |---|---|
   | **数据矛盾** | 同一指标/事实在不同页面有不一致的描述 |
   | **孤立页面** | 页面无任何入链（其他页面未引用它） |
   | **过期声明** | 页面中有明确时间标注的声明，距今超过8周且无后续更新 |
   | **空章节** | 页面中有章节标题但内容为空 |
   | **index 缺失** | wiki/ 目录下有页面但未在 index.md 中登记 |

3. **输出报告**
   ```
   ## Lint 报告 · {日期}

   ### 数据矛盾（{n} 处）
   - {页面A} 与 {页面B} 对 {指标} 的描述不一致：...

   ### 孤立页面（{n} 个）
   - {页面路径}

   ### 过期声明（{n} 处）
   - {页面路径}："{声明内容}"，最后更新 {日期}

   ### 其他问题
   ...

   ### 建议处理方式
   ...
   ```

4. **写入操作日志**
   - 在 `wiki/log.md` 追加 Lint 条目，记录发现问题数量和处理状态

**修复：** Lint 只输出问题，不自动修复。由用户决定哪些问题需要修复，再单独处理。
