# 维度五：知识锚定（Knowledge Grounding）

**核心问题**：这个 Skill 做出的判断，依据是什么？

**评分制度**：满分 10 分，基准按 Skill 类型不同而不同（见下方分类）。

---

## 第一步：判断 Skill 类型

在评分之前，先判断被审查 Skill 属于哪种类型。**不同类型对知识锚定的要求完全不同。**

### A 类：纯流程执行型（Knowledge Grounding 不适用）

特征：
- Skill 的职责是**收集参数 → 调接口 → 返回结果**
- 不需要做任何专业判断，只是流程编排
- 正确性由接口返回值决定，而非 LLM 的知识

典型例子：`contract-creation`（创建合同）、`contract-search`（查合同）、表单提交类、数据查询类

**处理方式**：标注 `N/A — 纯流程执行型 Skill，无需知识锚定`，维度五不计入综合评分。

---

### B 类：流程 + 专业判断混合型（重点审查）

特征：
- 核心流程是执行操作，但**某些步骤需要专业判断**
- LLM 被要求基于领域知识给出结论（风险评估、合规判断、建议生成等）
- 如果 LLM 的判断来自训练数据记忆而非实时知识库，结论可能过时或不准确

典型例子：`contract-review`（合同审查）——调后端预审接口是流程，但解释风险、给修改建议需要法律知识

**处理方式**：完整执行本维度所有检查点。

---

### C 类：纯知识问答型（重点审查）

特征：
- Skill 的核心价值就是回答专业领域问题
- 完全依赖 LLM 知识或外部知识库
- 没有知识锚定 = 用模型记忆替代权威来源

典型例子：`contract-query-skill`（合同问答）、法律咨询类、合规检查类

**处理方式**：完整执行本维度所有检查点，且权重更高。

---

## 检查点（B 类和 C 类 Skill 适用）

### 5.1 — 判断来源透明度（-3 分）

**核心问题**：Skill 给出的专业结论，用户能知道依据是什么吗？

检查项：
- Skill 的输出是否标注了判断依据（引用的条款、来源文档、规则 ID）？
- 若依赖后端接口，接口返回的风险项是否有可追溯的规则来源？
- 若依赖 LLM 判断，是否有"综合推理/基于训练数据"等明确标注，而非以"原文答复"口吻呈现？

**扣分标准**：
- 输出结论无任何来源标注：-2 分
- 依赖接口黑箱且不向用户说明结论来源：-1 分
- LLM 判断以"权威结论"口吻呈现（而非"推理建议"）：-1 分

---

### 5.2 — 知识库 / RAG 接入（-3 分）

**核心问题**：需要专业判断的地方，有没有实时知识来源？

检查项：
- 是否接入了领域知识库（法律法规库、合规规则库、行业标准库等）？
- 是否有 RAG 检索机制（向量检索、全文检索、结构化规则查询）？
- 知识库是否会更新（静态 prompt 嵌入 vs 动态检索）？

**扣分标准**：
- 需要专业判断但完全无知识库：-3 分
- 有知识库但覆盖范围不完整（如只有内部规则，无法律法规）：-1 分
- 知识库是静态写死在 prompt 里（版本过时风险）：-0.5 分

**特别说明**：若后端接口封装了规则引擎（如预审接口内置了合同审查规则库），这算作部分满足，但仍需评估：
1. 规则库的覆盖范围是否足够（只覆盖格式/结构规则，还是也覆盖法律合规性）
2. LLM 侧是否有能力对规则引擎的结果做解释和补充

---

### 5.3 — 知识时效性（-1 分）

**核心问题**：这个 Skill 的知识会过时吗？有没有机制保障时效性？

检查项：
- 涉及的法律法规/行业标准是否有明确的版本标注（如"依据 2021 年版《民法典》"）？
- 是否有版本过时的风险（法律修订、政策变化）？
- 是否有检查/提示知识版本的机制？

**扣分标准**：
- 涉及时效性强的领域（法律、金融监管、税务）但无任何版本标注：-1 分
- 有版本标注但超过 1 年未更新：-0.5 分

---

### 5.4 — 兜底策略（-2 分）

**核心问题**：当知识库没有答案时，Skill 怎么处理？

检查项：
- 是否有明确的兜底策略（如 RAG 无结果时降级为"综合推理"并标注）？
- 是否有"不知道"的出口（明确告知用户需要人工确认，而非硬猜）？
- 用 LLM 兜底时，是否有质量标注（区分原文答复 vs 推理答复）？

**扣分标准**：
- 无兜底策略，知识库无答案时 LLM 直接硬猜：-2 分
- 有兜底但无质量标注（用户无法区分答案可信度）：-1 分
- 有兜底且有质量标注：0 分

---

### 5.5 — 知识与操作的分层（-1 分）

**核心问题**：知识判断层和操作执行层是否分开了？

检查项：
- 专业判断（"这个条款有风险"）和操作执行（"提交预审"）是否在流程上分层？
- 用户是否有机会在"知识判断"结果出来后、"操作执行"之前做决策？
- 知识判断的结果是否影响了后续操作路径（而不是被忽略）？

**典型反模式**：
- 合同审查结果出来后，直接提交，用户无法基于审查结论决定是否修改再提交
- 风险判断和数据提交混在同一步骤，无法单独优化知识判断质量

**扣分标准**：
- 专业判断和操作执行完全耦合：-1 分
- 有分层但用户决策节点不明确：-0.5 分

---

## 评分示例

### contract-review（B 类：流程+专业判断混合型）

| 检查点 | 发现 | 扣分 |
|--------|------|------|

| 5.3 知识时效性 | 后端规则库版本不透明，Skill 文档无任何法律版本标注 | -1 |
| 5.4 兜底策略 | 后端返回 0 个风险项时，Skill 直接输出"未发现风险"，无 LLM 兜底检查 | -2 |

| **合计** | | **10 - 9.5 = 0.5/10** |

## 高分设计模式（供参考）

**模式一：规则引擎 + LLM 解释层**
```
后端规则引擎（结构化规则库）
    ↓ 返回命中规则ID + 风险等级
LLM（接收规则ID，从知识库检索规则原文）
    ↓ 生成人话解释 + 修改建议 + 法律依据
用户看到：风险项 + 法律条文引用 + 具体修改建议
```

**模式二：RAG 驱动的知识问答**
```
用户问题（补充审查事项 / 合同条款）
    ↓
向量检索（法律条文库 + 合同规范库 + 案例库）
    ↓ 返回相关文档片段
LLM（基于检索结果生成答案，标注出处）
    ↓
用户看到：答案 + [出处链接] + 可信度标注
```

**模式三：分层审查**
```
第一层：格式/结构审查（规则引擎，快速，高准确率）
第二层：法律合规审查（RAG + LLM，针对关键条款）
第三层：商业风险评估（LLM 推理，标注为"参考建议"）
```
