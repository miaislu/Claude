# 维度二：LLM/脚本边界（LLM vs Script Boundary）

**评分制度**：满分 10 分，基准 10 分，根据发现的职责错配扣分。

**核心判断原则**：

> **"能用 if/else 写清楚的逻辑，不该交给 LLM。"**
> LLM 的不可替代价值只有一件事：把人说的话转成结构化参数，以及把结构化数据转成人话。其余一切都应该是确定性代码。

---

## 脚本应承担的操作（LLM 做了要扣分）

以下操作由 LLM 执行是职责错配，每类按严重程度扣分：

### 状态管理类（-2 分/项）

**检查点 2.1 — 运行时变量存在 LLM 上下文**

运行时需要跨步骤传递的变量（ID、token、URL、用户选择结果等）靠 LLM 在对话上下文中"记住"，而没有写入持久化文件（session 文件、状态文件）。

识别方式：工作流中有类似"保存 xxx，用于后续步骤"的指令，但没有配套的"写入文件/读取文件"步骤。

严重后果：对话超时/中断 → 状态全丢，用户必须重头来；上下文变长后 LLM 容易"遗忘"早期变量值。

---

### 控制流类（-1.5 分/项）

**检查点 2.2 — 步骤跳转由 LLM 判断**

工作流的分支条件（"如果 A 则做 X，否则做 Y"）由 LLM 读文档后自行判断，而不是由代码状态机控制。

识别方式：SKILL.md 中的流程分支写成自然语言条件句，没有配套的脚本/状态机来强制执行步骤顺序。

---

**检查点 2.3 — 轮询循环由 LLM 控制**

需要重复执行某个操作直到满足条件（轮询接口状态、等待异步任务完成）的逻辑，完全由 LLM 维护循环计数和退出条件。

识别方式：工作流中有"每 X 秒轮询一次，最多 N 次"的描述，但没有配套脚本来执行 while 循环。

---

### 数据处理类（-1 分/项）

**检查点 2.4 — 枚举映射由 LLM 完成**

将用户输入的序号/选项映射到数据库 ID 或内部枚举值，完全由 LLM 凭记忆完成，没有脚本维护映射表。

典型案例：用户选择"第 1 和第 3 个清单"，LLM 需要把序号 [1, 3] 映射到数据库 id [101, 203]。

---

**检查点 2.5 — 数值统计由 LLM 完成**

需要精确计数（风险项数量、数组长度求和）的操作由 LLM 完成，而不是脚本执行 `count()`/`sum()`。

LLM 计数不可靠，尤其在嵌套数组场景下极易出错。

---

**检查点 2.6 — 字符串截断由 LLM 完成**

有明确字符数/字节数限制的字段（如"最多 500 字"）依赖 LLM 自觉控制长度，而没有脚本层面的 `text[:500]` 截断。

---

**检查点 2.7 — URL/字符串拼接由 LLM 完成**

构造接口 URL、Cookie、链接等纯字符串拼接操作由 LLM 完成。

---

### 接口调用类（-1 分/项）

**检查点 2.8 — 工具结果未标准化即传给 LLM**

接口返回的原始 JSON（尤其是三层以上嵌套）直接交给 LLM 解析，而没有脚本层将其提取为扁平化的干净结构。

识别方式：文档中出现 `data.data.someList[0].fieldName` 这类深层路径，LLM 需要自行访问。

风险：LLM 访问深层字段时字段名稍有偏差就会静默出错（如访问不存在字段返回 undefined 而非报错）。

---

**检查点 2.9 — 认证 token 构造由 LLM 完成**

SSO Cookie 构造、Bearer token 拼接等纯字符串操作由 LLM 完成，而不是脚本层完成后以环境变量形式传入。

---

## LLM 应承担的操作（做了是加分的，脚本无法替代）

以下操作 LLM 是不可替代的，应由 LLM 完成：

| 操作类型 | 说明 |
|---------|------|
| **自然语言意图识别** | 把"跳过"/"确认"/"不对换一个"等自然语言归一到枚举值 |
| **模糊输入参数提取** | 从用户自由表达中提取搜索关键词、文件路径、编号等结构化参数 |
| **输入类型判断** | 判断用户输入是"TP 开头的编号"还是"URL"还是"关键词" |
| **自由文本生成** | 将结构化数据（风险 DTO）转为自然语言描述（风险报告、修改建议） |
| **话术格式化** | 将系统状态（步骤完成、遇到错误）转为面向用户的自然语言提示 |

---

## 架构反模式识别（附加检查）

以下反模式若发现，在问题描述中注明对应的检查点：

| 反模式 | 对应检查点 | 典型表现 |
|--------|-----------|---------|
| LLM 当状态机 | 2.1 + 2.2 | 整个工作流是自然语言指令，没有任何持久化状态 |
| LLM 当轮询器 | 2.3 | "每 10s 查一次，直到状态变为完成" 写在 SKILL.md 主体中 |
| LLM 当计算器 | 2.5 | "统计 riskLevel=1 的项目数量" 无脚本辅助 |
| LLM 当 JSON 解析器 | 2.8 | 三层嵌套的响应结构让 LLM 自行 navigate |
