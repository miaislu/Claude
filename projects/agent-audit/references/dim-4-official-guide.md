# 维度四：官方 Skill 指南符合度（Official Guide Compliance）

**来源**：*The Complete Guide to Building Skills for Claude*（Anthropic 官方，2026年1月版）  
**评分制度**：满分 10 分，基准 10 分，根据不符合项扣分，加分项可补偿但上限 10 分。

---

## 扣分项

### YAML Frontmatter 合规性（最高扣 3 分）

**检查点 4.1 — name 字段格式**（-0.5 分）

- 必须是 kebab-case（小写字母、数字、连字符）
- 不能有空格、大写字母、下划线
- 应与文件夹名称一致

违反示例：`name: MySkill`、`name: my_skill`、`name: My Skill`

---

**检查点 4.2 — description 字段完整性**（-1 分）

根据官方指南，description 必须同时包含：
1. **做什么**（What it does）：一句话描述核心功能
2. **何时触发**（When to use it）：具体的触发短语枚举，用户实际会说的话

缺少任意一部分即扣分。

优秀格式示例：
```
description: 分析 Figma 设计文件并生成开发交接文档。
  触发词：上传 .fig 文件、要求设计规范、组件文档、design-to-code 交接。
```

---

**检查点 4.3 — description 触发词格式**（-0.5 分）

触发词应以**枚举列表**呈现（逗号分隔、中文顿号分隔或换行列举），而非散文句式。

散文式（较差）：`"当用户提到合同审查、审核或风险检查时触发"`  
枚举式（较好）：`"触发词：合同审查、合同预审、风险检查、审核合同"`

---

**检查点 4.4 — 安全限制违规**（-1 分/处，最多 -2 分）

Frontmatter 中的禁止项：
- XML 角括号（`<`、`>`）：安全限制，Frontmatter 出现在系统提示中
- Skill 名称包含 `claude` 或 `anthropic`（保留字）
- YAML 中的代码执行语法（使用安全 YAML 解析器，不会执行）

---

### 渐进式披露执行（最高扣 3 分）

**检查点 4.5 — SKILL.md 体积控制**（-1 分）

官方指南建议 SKILL.md 控制在 **5,000 词以内**。

评估方法：统计 SKILL.md 的字数（`wc -w`）或 token 数（中文约 1.5 token/字，英文约 1.3 token/词）。

超过 5,000 词的症状：把本应在 references/ 中的详细规范内联在主文件里。

---

**检查点 4.6 — 缺少步骤级 read 导航指令**（-1 分）

官方指南要求 references/ 文件是"LLM 按需导航发现"的知识节点（第三层渐进披露），而不是全量加载。

检查要点：工作流各步骤是否有明确的 `read references/xxx.md` 指令？
- ✅ 好的写法：`③ 模板识别（read references/template_identify_spec.md）`
- ❌ 差的写法：文件索引表放在文末，步骤中不提及何时读

没有导航指令，LLM 只能全量加载所有 references/ 文件，违反渐进披露原则。

---

**检查点 4.7 — 全量注入反模式**（-1 分）

Skill 的实现方式导致所有文档在第一步就全部加载进上下文（如 SKILL.md 中用 `read all references/` 的方式），而不是按步骤按需加载。

注意：检查点 4.6（缺导航指令）和 4.7（全量注入）通常同时出现，但 4.6 关注"文档结构设计"，4.7 关注"运行时注入行为"，分别扣分。

---

### 可测试性（最高扣 2 分）

**检查点 4.8 — 缺少触发测试用例**（-0.5 分）

官方指南建议提供：
- **Should trigger** 列表：哪些用户输入应该激活这个 Skill（至少 5 条）
- **Should NOT trigger** 列表：哪些用户输入不应激活（至少 3 条）

目的：防止触发漂移（过度触发或触发不足），每次迭代后可以快速验证。

---

**检查点 4.9 — 缺少功能测试用例**（-0.5 分）

应提供至少 3 条 Given/When/Then 格式的功能测试用例，覆盖核心路径：

```
Given: 用户上传一份对方模板合同
When: 执行预审流程
Then: 输出包含"对方模板"标记的风险报告，且红线数统计正确
```

---

**检查点 4.10 — 缺少基线对比数据**（-1 分）

官方指南建议提供"有 Skill vs 无 Skill"的基线对比：
- token 消耗对比
- 完成任务所需对话轮数对比
- API 调用次数对比（成功率）

缺少基线数据意味着无法量化 Skill 的价值，也无法在迭代时判断是否退化。

---

### 指令可遵循性（最高扣 2 分）

**检查点 4.11 — 关键约束未置顶**（-0.5 分）

官方指南建议：最重要的约束（不可违反的规则）应放在 SKILL.md 最顶部，使用明确的标题（`## ⛔ 核心约束`、`## CRITICAL`）突出显示。

关键约束埋在文档中间或末尾，LLM 可能在上下文很长时漏读。

---

**检查点 4.12 — 指令语言模糊**（-0.5 分）

官方指南明确指出"模糊语言是 LLM 不遵守指令的首要原因"。

模糊示例（差）：`"确保适当验证所有输入"`  
具体示例（好）：`"调用 create_project 前，必须验证：project_name 非空、至少一个成员、start_date 不早于今天"`

---

**检查点 4.13 — skill.manifest 缺失或不完整**（-1 分）

- 无 skill.manifest 文件：-0.5 分
- skill.manifest 中列出的文件与实际文件不匹配：-0.5 分
  - 文件存在但不在 manifest 中
  - manifest 列出了不存在的文件（悬空引用，与检查点 1.4 重叠时不重复扣分）

---

## 加分项（上限 +1 分）

**加分项 4.A — 完善的错误处理说明** +0.5 分

每种可能的失败场景（接口超时、认证失败、文件格式不支持等）都有明确的错误提示格式和处理路径。

**加分项 4.B — 版本 changelog 或更新记录** +0.5 分

SKILL.md 或单独文件中记录了版本迭代历史，说明每个版本修复了什么问题。这有助于维护者追溯问题根因。

## 评分示例

| 发现 | 扣分 |
|------|------|
| description 触发词是散文式而非枚举式 | -0.5（检查点 4.3） |
| references/ 文件无步骤级导航指令 | -1（检查点 4.6） |
| 无触发测试用例 | -0.5（检查点 4.8） |
| 无功能测试用例 | -0.5（检查点 4.9） |
| 无基线对比数据 | -1（检查点 4.10） |
| 关键约束未置顶 | -0.5（检查点 4.11） |
| log_spec.md 在 manifest 中缺失 | -0.5（检查点 4.13） |
| 有完善的错误处理（各步骤均有错误提示） | +0.5（加分 4.A） |
| **合计** | **10 - 4.5 + 0.5 = 6** |

## 快速自查清单（Pre-distribution Checklist）

在输出审查结果前，用此清单快速过一遍：

- [ ] name 是 kebab-case，无空格无大写
- [ ] description 包含"做什么"和"触发词枚举"
- [ ] 无 XML 角括号
- [ ] SKILL.md < 5,000 词
- [ ] 各步骤有 `read` 导航指令
- [ ] 至少 5 条 Should trigger 用例
- [ ] 至少 3 条 Should NOT trigger 用例
- [ ] 至少 3 条 Given/When/Then 功能测试
- [ ] 关键约束在顶部用 ⛔/CRITICAL 标题
- [ ] skill.manifest 存在且文件清单完整
- [ ] 无悬空引用（所有 `read` 的文件都存在）
