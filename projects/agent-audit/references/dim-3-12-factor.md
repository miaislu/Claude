# 维度三：12-Factor Agents 符合度

**来源**：https://github.com/humanlayer/12-factor-agents  
**评分制度**：满分 12 分，每个 Factor 1 分。  
**评级**：✅ 符合（1分）/ ⚠️ 部分符合（0.5分）/ ❌ 违反（0分）/ N/A 不适用（不计入分母）

---

## 12 个 Factor 逐条检查

### F1 — 自然语言 → 工具调用（Natural Language to Tool Calls）

**核心原则**：LLM 的职责是把自然语言翻译成结构化工具调用参数，确定性代码负责接管执行。

**检查要点**：
- LLM 输出结构化参数后，是否有确定性代码负责执行？
- 工具的参数粒度是否合理（太粗 = LLM 做了不该做的事；太细 = 过度拆分）？
- LLM 是否还在"翻译"之后继续负责执行（如调用接口、处理响应）？

**典型违反**：LLM 既生成参数，又执行接口调用，又处理响应，没有明确的"LLM 停止，代码接管"边界。

---

### F2 — 掌控你的提示词（Own Your Prompts）

**核心原则**：Prompt 是一等公民，应显式管理、版本化、可测试，不依赖框架黑箱。

**检查要点**：
- Prompt（SKILL.md + reference 文件）是否有版本号/变更记录？
- 各 reference 文件是否有版本对齐标记（对应哪个主 Skill 版本）？
- Prompt 内容是否可以被独立测试（单条规则可验证）？
- 是否有多处分散的 Prompt 片段（增加了维护成本和矛盾风险）？

**典型违反**：SKILL.md 是 V21，但 references/ 文件没有版本号，不知道哪个版本对应哪个。

---

### F3 — 掌控上下文窗口（Own Your Context Window）

**核心原则**：精细控制传给 LLM 的内容，按需注入，不全量加载，完成的步骤上下文应被裁剪。

**检查要点**：
- 是否所有文档在 Skill 启动时一次性全量注入？
- 不同步骤是否只加载该步骤需要的 reference 文件？
- 随着对话进行，上下文是否无限增长还是有裁剪机制？
- 步骤⑦（最复杂的任务）执行时，上下文是否已经很大？

**评估方法**：
1. 列出所有 reference 文件大小（token 估算：中文字符约 1.5 token/字）
2. 判断是否全量注入：若 SKILL.md 中没有按步骤的 `read` 指令，视为全量注入
3. 计算最坏情况的上下文大小（初始注入 + 对话历史）

**参考阈值**：单步 reference 注入 < 2k token 为良好；全量注入 > 5k token 为问题。

---

### F4 — 工具是结构化输出（Tools Are Structured Outputs）

**核心原则**：工具调用的本质是 LLM 输出结构化 JSON；工具结果也应被标准化后再传给 LLM。

**检查要点**：
- 接口响应是否被脚本层提取为扁平化、干净的结构后再传给 LLM？
- LLM 是否需要访问三层以上嵌套的 JSON 路径（如 `data.data.list[0].field`）？
- 工具调用的参数 schema 是否明确定义？

**评估方法**：在所有文档中搜索形如 `data.data.xxx` 或 `response.data.list[0]` 的深层路径引用，每找到一处即为问题信号。

---

### F5 — 统一执行状态与业务状态（Unify Execution State and Business State）

**核心原则**：执行状态（在哪一步）和业务状态（收集了哪些数据）要统一持久化，不分离存储，不靠 LLM 记忆。

**检查要点**：
- 是否有 session 文件、状态文件来持久化运行时变量？
- 关键变量（文件 ID、任务 ID、用户选择结果等）是否在步骤执行后写入文件？
- 是否有"中断后恢复"的能力（依赖 F5 的实现）？

**识别方法**：在工作流中找所有"保存 xxx 用于后续步骤"的描述，检查是否有对应的写文件操作。若只在 LLM 上下文中"记忆"，即为违反。

---

### F6 — 支持启动/暂停/恢复（Launch, Pause, Resume）

**核心原则**：Agent 应支持在等待外部事件（用户输入、异步任务完成）时中断，并在收到信号后从断点恢复。

**检查要点**：
- 如果用户在等待期间关闭了对话窗口，重新打开后能否继续？
- 长时间等待（轮询、等待用户回复）期间是否会丢失状态？
- 是否有 webhook/回调机制，或只有主动轮询？

**注意**：若 Skill 是纯同步、单轮对话型（无需等待异步操作），标注 `N/A — 无异步操作`，不扣分。

---

### F7 — 用工具联系人类（Contact Humans with Tool Calls）

**核心原则**：需要人工介入时，Agent 应输出结构化的 `request_human_input` 工具调用，然后保存状态并断开循环等待 webhook，而不是在循环里等待用户回复。

**检查要点**：
- 用户等待节点（⏸️）是通过结构化工具调用触发，还是 LLM 发完消息后在循环内等待？
- 是否有明确的"保存状态 → 断开循环 → 等待 webhook 恢复"流程？

**现实考量**：在 Claude Skill 场景中，完整的 F7 实现需要平台支持，很多 Skill 只能做到"发消息等回复"。这种情况给⚠️（部分符合），而非❌。

---

### F8 — 掌控控制流（Own Your Control Flow）

**核心原则**：控制流（步骤跳转、分支、循环）应由业务代码掌控，不交给 LLM 自行决定。

**检查要点**：
- 工作流步骤是"写给 LLM 看的自然语言指令"还是"机器执行的状态机"？
- 步骤跳转条件（if A then X else Y）是否在代码中实现？
- 是否存在"LLM 读懂所有文档，然后决定接下来做什么"的模式？
- 轮询循环是否由脚本控制（while + sleep）？

**最常见的违反**：整个 SKILL.md 就是一份"自然语言操作手册"，没有任何状态机或调度代码。

---

### F9 — 错误压缩进上下文（Compact Errors Into Context）

**核心原则**：错误发生时，将错误信息格式化为结构化事件后纳入上下文，让 LLM 基于结构化错误信息决策（降级/重试/终止），而不是让 LLM 自行解析原始 HTTP 响应或错误信息。

**检查要点**：
- 接口失败时，是原始 HTTP 响应/错误码直接传给 LLM，还是有脚本层将其转为结构化错误事件？
- 错误处理路径是否明确（每种错误对应什么行为）？
- 是否存在降级策略（失败 → 静默使用默认值继续）？

**评分说明**：有降级策略但错误未结构化 → ⚠️；完全没有错误处理 → ❌；有结构化错误 + 降级 → ✅。

---

### F10 — 小而专注的 Agent（Small, Focused Agents）

**核心原则**：单个 Agent（Skill）控制在 3-20 步，上下文随步骤增长会让 LLM 失焦；超出能力边界的任务应拆分为多个 Agent 协作。

**检查要点**：
- 工作流步骤数量是否在 3-20 之间？
- 步骤是否职责单一（一步做一件事）？
- 最复杂的步骤（通常是最后的输出步骤）执行时，上下文是否已经很大？
- 该 Skill 是否试图同时处理太多不同类型的任务？

---

### F11 — 多渠道触发与响应（Trigger from Anywhere, Respond in Kind）

**核心原则**：Agent 应能从多种渠道触发（IM 消息、webhook、定时任务、API 调用），并在用户所在渠道响应。

**检查要点**：
- 触发渠道是否只有一种，还是支持多种？
- 响应是否在用户触发的同一渠道？
- 是否支持被其他 Agent 或自动化流程触发（outer loop 模式）？

**注意**：企业内部 Skill 通常只在单一 IM 平台运行，单渠道完整闭环可给⚠️（部分符合）而非❌。

---

### F12 — Agent 是无状态 Reducer（Stateless Reducer）

**核心原则**：Agent 的每次执行模型应为 `(state, event) → new_state`，即纯函数式设计，使 Agent 可被测试、复现和 debug。

**检查要点**：
- 是否有明确的 state 对象（写入文件的 session 状态）？
- 每次执行是否产生新的 state（而不是在可变的对话上下文中累积）？
- 是否可以通过"构造任意 state + event"来测试 Agent 的行为？
- 是否支持 replay（相同 state + event → 相同 new_state）？

**与 F5 的关系**：F5 关注"状态持久化存在哪"，F12 关注"执行模型是否是纯函数式"。两者互补，F5 是 F12 的前提。
