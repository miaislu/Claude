## 执行摘要

contract-search 是四个合同 Skill 中**最轻量、最单纯**的一个——纯流程执行型，职责清晰（查询合同数据并展示），CLI 封装了所有接口调用，LLM 只负责意图识别和结果格式化。文档完整度适中，但存在几处值得关注的问题：版本检查逻辑低效（每次激活都全量安装而非版本对比）、manifest 中不含 assets/ 目录的三个文件，以及完全缺失测试规范。

综合来看，这是一个可以直接交付的 Skill，主要缺口在于文档的维护规范性，无架构层面问题。

### 🟠 P1 — 稳定性问题（1 项）

---

**P1-1　CLI 版本检查每次全量安装，无版本对比逻辑**

- **位置**：`SKILL.md` "前置检查：确保 CLI 最新" 节
- **问题**：当前脚本直接执行 `npm list` 查看是否安装，如未安装则运行 `npm install -g @cap/skills-legal@latest`。但没有像 contract-creation 那样先对比本地版本和远程最新版本，只要"安装了"就跳过更新——如果本地安装的是旧版，不会触发更新。极端情况下用户本地的旧版 CLI 缺少新命令会导致运行失败，而 Skill 不会提示。
- **修复**：采用 contract-creation 的版本对比脚本（先 `npm list -g` 拿本地版本，再 `npm view @latest version` 拿远程版本，不一致才安装）。
- **来源维度**：文档严谨性（检查点 1.8）

---

### 🟡 P2 — 一致性问题（2 项）

---

**P2-1　assets/ 三个枚举文件在 manifest 中缺失**

- **位置**：`skill.manifest` vs 实际文件
- **问题**：`assets/contractStatus.md`、`assets/archiveStatus.md`、`assets/auditFlowStatus.md` 三个文件实际存在于 zip 包中，但 `skill.manifest` 的 files 清单中没有列出它们。manifest 用于验证文件完整性，缺失意味着这三个文件无法被完整性校验覆盖，分发时若文件被篡改也无法检出。
- **修复**：将三个 assets 文件加入 manifest，重新生成 skill-id 哈希。
- **来源维度**：官方 Skill 指南（检查点 4.13）

---

**P2-2　缺少测试规范（触发测试、功能测试、基线对比）**

- **位置**：整个 Skill 包，无 tests/ 目录
- **问题**：无 Should trigger / Should NOT trigger 列表；无 Given/When/Then 功能测试用例；无有/无 Skill 的基线对比。触发词较少（8 个），有过宽触发的风险（如"查合同"是否会与 contract-query-skill 的问询型合同查询重叠）。
- **修复**：新增 `tests/trigger-tests.md`，至少包含：应触发场景 5 条（"查我的合同列表"、"帮我查合同 CO260312000224"等）、不应触发场景 3 条（"合同怎么归档"等应走 contract-query-skill）。
- **来源维度**：官方 Skill 指南（检查点 4.8/4.9）

---

### 🟢 P3 — 优化项（3 项）

---

**P3-1　description 缺少"不适用"场景说明**

- **位置**：`SKILL.md` frontmatter description 字段
- **问题**：description 只说明了"支持什么"，没有像 contract-creation 那样列出"不支持什么"（如"不支持合同内容问答，不支持合同发起"）。触发词"法务BP"与 contract-query-skill 有重叠——同一个词两个 Skill 都会触发，用户可能收到混乱的响应。
- **建议**：在 description 末尾加 `不适用于：合同流程问答（请使用 contract-query-skill）、合同发起操作。`
- **来源维度**：官方 Skill 指南（检查点 4.2）

---

**P3-2　references/ 文件在 SKILL.md 中只有静态索引，无步骤级导航指令**

- **位置**：`SKILL.md` "参考文档"节
- **问题**：SKILL.md 末尾有参考文档静态列表，但没有说明"查询合同列表时 read references/queryContract.md"这样的步骤级导航。LLM 可能在不需要的时候加载所有 reference 文件，或在需要时遗漏。
- **建议**：在各场景说明节（场景一/二/三/四）的开头加 `（接口规范：read references/xxx.md）` 导航提示。
- **来源维度**：官方 Skill 指南（检查点 4.6）

---

**P3-3　常用状态码列表不完整，用户需要自行查找枚举文件**

- **位置**：`SKILL.md` 场景二"常用状态码"
- **问题**：SKILL.md 中只列了 4 个常用状态码（草稿/审批中/已生效/盖章完成），但 `assets/contractStatus.md` 里有完整枚举。用户查询特定状态时（如"已作废"、"已终止"），SKILL.md 无法给出状态码，需要 LLM 自行读取 assets 文件。没有导航指令指向这个文件，LLM 可能遗漏。
- **建议**：在状态码说明处加注 `（完整枚举见 assets/contractStatus.md）` 导航提示。
- **来源维度**：官方 Skill 指南（检查点 4.6）+ 文档严谨性

## 架构层建议

本 Skill 的 LLM/脚本分工整体合理，无系统性架构调整建议。

contract-search 是四个 Skill 中架构最干净的一个：
- CLI 封装了全部 SSO、接口调用、响应解析、表格格式化
- LLM 只负责两件事：从用户自然语言中提取查询参数（MIS、合同号、状态等），以及将 CLI 输出格式化为对话回复
- 这是标准的"LLM 作翻译器"模式，完全符合 12-Factor F1 的设计意图

## 知识锚定

**Skill 类型：A 类 — 纯流程执行型**

contract-search 的职责是查询合同系统中的结构化数据并展示，不做任何专业判断。返回的是事实（合同状态、审批人、生效日期），不是解释或建议。维度五不适用，不计入综合评分。

## 值得保留的优点

1. **错误码文档完整**：`errorCode 999001 / 300000` 均有含义说明和处理步骤，是四个 Skill 中错误处理文档最完整的。
2. **风控和限制明确声明**：超时控制、重试策略、限流、并发控制、Token 缓存有效期——这些工程约束被清晰列出，其他 Skill 均无此声明。
3. **两个 ClientId 分工清晰**：合同系统（`039147573f`）和法务系统（`com.sankuai.it.jwl.app`）的用途明确，与 contract-review 的 SSO 规范保持一致。
4. **`--raw` 调试参数**：提供原始 JSON 输出选项，方便排查接口异常，是贴近开发者需求的设计。
