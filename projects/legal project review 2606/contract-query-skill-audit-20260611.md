# contract-query-skill Skill 审查报告

**版本**：V25　**审查日期**：2026-06-11
**评审维度**：文档严谨性 × LLM/脚本边界 × 12-Factor Agents × 官方 Skill 指南 × 知识锚定
**审查工具**：agent-audit v1.1.0

---

## 执行摘要

contract-query-skill 是四个 Skill 中**设计最复杂、功能最全面**的一个，也是在知识锚定维度表现最好的——它明确接入了 MBookLM RAG 知识库（36份文档），有清晰的三级知识使用优先级（结构化知识→RAG→综合推理），并强制要求每条回复附带出处链接和原文引用。

但这个 Skill 的复杂度也带来了几个显著问题。最值得关注的是**版本号自相矛盾**（frontmatter 是 2.8.0，版本检查脚本写死了 CURRENT="V21"，metadata 写的是 V25）；以及**用户信息获取逻辑中有硬编码的个人 UID**，暴露了维护者身份信息并且导致该功能只对单人有效。SKILL.md 体积高达 1271 行，远超官方建议的 5000 词上限，这是最严重的结构问题。

综合来看：知识层设计是标杆，但文档膨胀和代码质量需要整改。

## 问题清单

### 🔴 P0 — 必须修复（1 项）

---

**P0-1　用户信息获取逻辑硬编码了维护者个人 UID，部门链功能对其他用户完全失效**

- **位置**：`SKILL.md` "用户信息获取方法" 节（第 1027 行附近）
- **问题**：获取用户完整部门链的代码中写死了：
  ```bash
  if [ "<SENDER_UID>" = "3042057488" ]; then
    DEPT_RAW=$(oa-skills calendar-rest getUserInfo --raw 2>/dev/null)
  fi
  ```
  `3042057488` 是维护者 liqing76 的个人 UID。这导致：① 部门链功能只对 liqing76 一人有效，其他所有用户部门链留空；② 维护者个人 UID 硬编码在 Skill 文档中，属于内部身份信息暴露；③ 代码注释自己也承认了这个问题（"注意：此命令获取的是当前登录用户（liqing76）的信息"）。
- **修复**：若 `calendar-rest getUserInfo` 接口无法通过 SENDER_UID 查询任意用户的部门，则应将部门链字段设为"始终留空"并移除该条件判断，或接入支持按 empId 查询的接口（如通过 `queryUserIdentityByUid` 拿到 MIS 后再查部门）。
- **来源维度**：文档严谨性（隐私/安全）

---

### 🟠 P1 — 稳定性问题（3 项）

---

**P1-1　SKILL.md 体积高达 1271 行，严重违反 F3 上下文窗口管理**

- **位置**：`SKILL.md` 整体
- **问题**：整个 Skill 只有一个 1271 行的 SKILL.md 文件，没有任何 references/ 子目录。节点 1-7 的操作指引（合同发起、审批、盖章、归档、作废、借阅、权限）全部内联在主文件中，加上附录 A（管理员操作）、用户反馈收集（带完整 shell 脚本）、RAG 兜底节，每次调用 Skill 时 LLM 都需要将全部 1271 行加载进上下文。这使上下文在第一轮就达到约 12k+ token，后续对话历史叠加后 LLM 注意力严重分散。
- **修复**：将节点 1-7 各自拆分为 `references/node-X-xxx.md`，SKILL.md 只保留触发规范、核心约束、使用原则和导航指令。目标：SKILL.md 降至 200 行以内，每次只加载当前问题相关的节点文件。
- **来源维度**：12-Factor F3 + 官方 Skill 指南（检查点 4.5）

---

**P1-2　版本号在三处出现三个不同的值**

- **位置**：`SKILL.md` frontmatter `skill.version`、版本检查脚本、`metadata.skillhub.version`
- **问题**：
  - frontmatter `skill.version: 2.8.0`
  - 版本检查脚本：`CURRENT="V21"`（这是写死在脚本里的旧值，明显未同步更新）
  - `metadata.skillhub.version: "V25"`
  
  三处版本号格式和数值均不一致。版本检查脚本里的 `CURRENT="V21"` 尤其危险——如果广场上的版本是 V25，脚本会每次提示"有新版 V25"，形成永久误报噪音；更严重的是，若用户基于此判断 Skill 是否需要更新，会得到错误结论。
- **修复**：统一版本号为 `V25`，删除 `skill.version: 2.8.0`（这个 semver 格式和 Skillhub 的 V-prefix 格式混用增加混乱），版本检查脚本改为 `CURRENT="V25"`。
- **来源维度**：文档严谨性（检查点 1.1）+ 12-Factor F2

---

**P1-3　反馈写入逻辑复杂，rowId 依赖临时文件跨步骤传递，中断即失效**

- **位置**：`SKILL.md` "用户反馈收集" 节
- **问题**：写入多维表格后，用时间戳查询刚写入的行拿 rowId，存入 `/tmp/contract_last_row_id.txt`；用户反馈时从该文件读取 rowId 再 update。这个方案有两个问题：① 临时文件依赖机器状态，若 /tmp 被清理或进程隔离，rowId 丢失，用户反馈无法关联到原记录；② 高并发场景（同一用户两个对话窗口同时提问）时两个进程会互相覆盖 rowId 文件，导致反馈写入到错误的记录。
- **修复**：rowId 应存入会话级别的状态（如 session 变量或 context 变量），而非全局临时文件；或将 rowId 作为结构化字段嵌入输出上下文中，由 LLM 在对话轮次中持有。
- **来源维度**：12-Factor F5（状态管理）

---

### 🟡 P2 — 一致性问题（3 项）

---

**P2-1　frontmatter 中 description 和 skill.description 重复定义，内容一致但冗余**

- **位置**：`SKILL.md` frontmatter 第 1-18 行
- **问题**：顶层 `description` 字段和 `skill.description` 字段内容几乎完全一致，构成冗余定义。官方格式只需顶层 `description` 即可，`skill:` 嵌套块是非标准扩展字段。这会让维护者困惑"改哪一个"，时间一长两处会出现分歧。
- **修复**：删除 `skill:` 嵌套块（或仅保留非重复的字段如 `creator`、`triggers`），统一使用顶层 `description`。
- **来源维度**：文档严谨性（检查点 1.3 旧版残留）+ 官方 Skill 指南

---

**P2-2　版本检查脚本的 CURRENT 版本与 metadata 版本不同步**

（已在 P1-2 中描述，此处作为独立的文档一致性问题记录。）

- **位置**：`SKILL.md` 版本检查脚本 vs metadata 节
- **简述**：脚本 `CURRENT="V21"` 与 `metadata.skillhub.version: "V25"` 矛盾，且备份文件 `SKILL.md.v2.6.0.bak` 存在于 zip 包中说明有历史版本残留。
- **修复**：同步版本号，删除 `.bak` 备份文件（不应出现在发布包中）。
- **来源维度**：文档严谨性（检查点 1.1/1.3）

---

**P2-3　缺少测试规范（触发测试、功能测试）**

- **位置**：整个 Skill 包
- **问题**：无 Should trigger / Should NOT trigger 列表——contract-query-skill 的触发词极度宽泛（包含"合同"一词的所有问题），需要测试用例来界定触发边界（"帮我查合同 CO123"应该走 contract-search，不应走这里）。无功能测试用例。
- **修复**：新增 `tests/trigger-tests.md`，重点测试"操作要求 vs 问询"的边界（如"帮我发起合同"应路由出去，"合同怎么发起"应命中本 Skill）。
- **来源维度**：官方 Skill 指南（检查点 4.8/4.9）

---

### 🟢 P3 — 优化项（3 项）

---

**P3-1　附录 A（管理员操作）与普通员工内容混在同一文件，结构层次不清**

- **位置**：`SKILL.md` 附录 A（第 1187 行起）
- **问题**：附录 A 的管理员操作指引（A.1–A.6）与节点 1-7 的普通员工操作内联在同一个文件，每次调用都会加载。管理员操作频率极低但体积大，增加了无效上下文负担。
- **建议**：将附录 A 拆分为 `references/admin-guide.md`，在 SKILL.md 中加入"若用户提及管理员操作，read references/admin-guide.md"的条件导航。
- **来源维度**：官方 Skill 指南（渐进披露）

---

**P3-2　用户反馈写入脚本体积过大（约 100 行），占用 SKILL.md 大量篇幅**

- **位置**：`SKILL.md` "用户反馈收集" 节（第 969 行起）
- **问题**：完整的 shell 脚本（addData、queryTableData、updateData 命令及其 JSON 构造逻辑）内联在 SKILL.md 中，约占 100 行。这部分是技术实现细节，不是 LLM 需要理解的业务逻辑，每次都加载进上下文是浪费。
- **建议**：将写入脚本封装为 `scripts/feedback-write.sh`，SKILL.md 中只保留调用方式和参数说明。
- **来源维度**：官方 Skill 指南（检查点 4.5）+ 12-Factor F3

---

**P3-3　RAG 知识库的 36 份文档列表写在 SKILL.md 主体中，无时效标注**

- **位置**：`SKILL.md` RAG 兜底节"知识库覆盖的文档（36份）"
- **问题**：36 份文档的列表写在主文件中，但没有"最后更新于 YYYY-MM"的时效标注。合同系统的操作手册和规范随系统迭代会更新，知识库中可能存在过时文档。用户无法判断答案是否基于最新版本文档。另外，文档数量"36份"是否仍然准确也值得核查。
- **建议**：在知识库文档列表标注最后同步日期（如"知识库最后更新：2025-12"），并建立季度更新机制。
- **来源维度**：知识锚定（检查点 5.3）

## 架构层建议

**LLM 的职责分配整体合理**，问题主要在上下文管理和状态管理：

```
【当前】
SKILL.md 1271 行一次性全量加载（12k+ token）
  ├─ 节点1-7 操作指引（约 700 行）
  ├─ 附录 A 管理员操作（约 200 行）
  ├─ RAG 兜底（约 80 行，含脚本）
  └─ 反馈写入（约 100 行，含脚本）

【建议】
SKILL.md（目标 150 行）
  ├─ 触发规范
  ├─ 核心约束
  ├─ 导航指令（按问题类型 read 对应文件）
  └─ 联系方式速查 + 系统入口速查

references/node-1-start.md     ← 合同发起（按需加载）
references/node-2-approval.md  ← 合同审批（按需加载）
... （节点 3-7 各一个文件）
references/admin-guide.md      ← 附录 A（极少加载）
scripts/feedback-write.sh      ← 反馈写入脚本
```

改造后每次实际加载约 1-2k token（SKILL.md 主体 + 单个节点文件），是当前的 1/6。

## 知识锚定分析

**Skill 类型：C 类 — 纯知识问答型**

这是四个 Skill 中唯一真正需要知识支撑的问答型 Skill，也是知识锚定做得最好的一个。

**5.1 判断来源透明度：3/3**
强制五段式回复（原文答复/综合推理标注 + 出处链接 + 出处原文 + 推理依据），且有三门禁自查（出处链接不填不能发送），是四个 Skill 中唯一做到"每条回复有可追溯来源"的设计。

**5.2 知识库/RAG 接入：2/3**
接入了 MBookLM 知识库（ID: f4689632-38c2-461a-876d-8201525a5574），覆盖 36 份文档，有三级优先级（结构化知识→RAG→综合推理）。扣 1 分原因：知识库覆盖的是内部操作手册类文档，不包含外部法律法规（如《民法典》合同章）；对于"电子签有没有法律效力"这类问题，只能靠内部 FAQ 回答而非法律原文。

**5.3 知识时效性：0.5/1**
RAG 知识库中的文档无时效标注，合同系统迭代后操作手册可能过时，用户无从判断。扣 0.5 分。

**5.4 兜底策略：2/2**
有明确的三级兜底（结构化知识→RAG→综合推理+标注），且 RAG 无结果时还引导用户在海螺系统内自行尝试，最后才提 TT 工单。全覆盖。

**5.5 知识与操作分层：0/1**
本 Skill 是纯问答型，无操作执行，N/A，不扣分。

**维度五总分：7.5/10**

## 值得保留的优点（应向其他 Skill 推广）

1. **三级知识优先级**：结构化知识 → RAG → 综合推理，且每级有明确的触发条件和质量标注，是知识管理的最佳实践。
2. **强制出处引用**：三门禁自查（发回复前必须有出处链接）从机制上杜绝了无依据的"通用知识回答"，contract-review 应参考此设计。
3. **用户反馈闭环**：自动写入多维表格 + 用户主动反馈更新，形成了知识质量的持续改进机制，是四个 Skill 中唯一有反馈飞轮的设计。
4. **权限边界清晰**：普通员工操作（节点1-7）和管理员操作（附录A）有明确分层，涉及管理员操作时主动告知权限限制。
5. **路由设计**：操作要求 vs 问询的判断原则清晰，路由表明确（不同操作意图对应不同 Skill）。
