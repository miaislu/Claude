# Contract-Review Skill 综合评审报告

**版本**：V21　**评审日期**：2026-06-11　**评审维度**：文档严谨性 × 编排设计 × 12-Factor Agents × 官方 Skill 指南 × 知识锚定

---

## 执行摘要

Contract-review 是四个合同 Skill 中**工程成熟度最高**的一个——故障被显式标注、降级策略完整、SSO 规范清晰。但五轮评审共同指向两个根本性问题，且方向相反：

> **LLM 在它不擅长的地方（流程控制、状态管理）承担了过多，在它最擅长的地方（合同语义理解、法律推理）却几乎缺席。**

第一个问题：状态管理、控制流、轮询循环——这些本该由确定性脚本承担的职责，全部压在 LLM 的对话上下文里。第二个问题也同样严重：审查这件事的核心价值——理解条款语义、发现跨条款风险、给出法律依据、生成可操作建议——全部被外包给了一个黑箱预审接口，LLM 只负责把接口返回的 JSON 格式化成 Markdown，这是对 LLM 能力的根本性误用。

第五轮引入知识锚定视角，揭示了这个"职责倒置"的全貌：把合同语义理解交给了规则引擎黑箱，把排版工作留给了 LLM。

## 问题清单与优先级

优先级定义：
- 🔴 **P0 — 阻断级**：会导致静默错误或结果不可信，必须在下一版本修复
- 🟠 **P1 — 稳定性**：导致行为不可预测，影响正常使用
- 🟡 **P2 — 一致性**：文档内部矛盾，执行结果可能随模型/上下文漂移
- 🟢 **P3 — 优化项**：不影响正确性，但影响成本/体验/可维护性

---

### 🔴 P0 — 必须修复（4 项）

---

**P0-1　billId 换取接口入参两版本并存，高并发下有结果串号风险**

- **位置**：`pre_review_submit_spec.md` 主体 vs 附录
- **问题**：主体正文写入参为 `{"auditBillNumber": "..."}` 精确查询；附录写入参为 `{"startTime":null, "endTime":null, "pageNo":1, "pageSize":10}` 全量分页查询。后者在多用户并发时会取到其他人的审查单，导致 billId 张冠李戴，轮询到别人的结果。
- **修复**：删除附录版本，唯一保留精确查询入参，并加注"必须传 auditBillNumber，禁止不带条件分页查询"。

---

**P0-2　风险数量统计字段名错误，导致红线数永远为 0**

- **位置**：`output_spec_pre_review.md` 数据来源说明
- **问题**：统计逻辑写的是 `累计每条 auditItemResults 数组的长度之和`，但接口返回的 DTO 字段名是 `results`，不存在 `auditItemResults` 字段。LLM 访问不存在的字段会静默返回空/0，导致用户看到"红线风险：0 项"的假阴性结果。
- **修复**：将所有出现 `auditItemResults` 的地方改为 `results`，并补充字段路径示例：`item.results[].contents`。

---

**P0-3　序号→清单 id 映射由 LLM 完成，存在幻觉风险**

- **位置**：`checklist_query_spec.md` + `interaction_templates.md` 步骤④
- **问题**：展示给用户的是序号（1、2、3），用户回复"1,3"后，需要把序号映射到数据库的 `checklist.id`（如 `[1, 203]`）。这个映射目前完全由 LLM 凭记忆完成。LLM 若记错或幻构 id，会提交错误的清单组合，而用户和系统均无感知。
- **修复**：脚本在获取清单列表后生成 `checklist_map = {1: 101, 2: 203, ...}` 并写入 session 文件；LLM 只提取用户选择的序号数组，脚本负责最终映射。

---

**P0-4　全部运行时状态存在 LLM 上下文中，对话中断即全部丢失（12-Factor F5/F12 违反）**

- **位置**：架构层，无对应文件
- **问题**：`s3UUID`、`traceId`、`auditBillNumber`、`billId`、`templateCode`、`templateVersion`、选中的 checklist ids——这些值全靠 LLM 在对话上下文中"记住"。对话超时、app 切换、网络中断均导致状态全丢，用户只能重新上传文件走完整流程。上下文变长后还存在 Lost-in-the-Middle 风险，LLM 可能遗忘早期步骤存储的值。
- **修复**：引入 session 状态文件（参考 contract-creation 的 `state_file` 机制），每步执行前读取、执行后写回。LLM 不持有任何运行时变量，只读取 session 文件中的值。

---

### 🟠 P1 — 稳定性问题（6 项）

---

**P1-1　控制流完全由 LLM 驱动，步骤顺序无机器保障（12-Factor F8 违反）**

- **问题**：SKILL.md 里的 8 步工作流是"写给 LLM 看的自然语言指令"，而不是机器执行的状态机。步骤顺序、跳转条件、错误路径，全靠 LLM 读懂并遵守文档。随着对话上下文增长，LLM 有概率跳步或乱序执行。
- **修复**：业务代码维护步骤枚举和转移表；LLM 每轮执行后只输出下一步的工具调用，控制流跳转由代码决定，不由 LLM 自行判断"该做什么了"。

---

**P1-2　轮询循环由 LLM 控制（步骤⑥），不稳定且资源浪费**

- **问题**：预审提交后的轮询（最长 6 分钟，36 次）完全由 LLM 驱动——LLM 决定等待间隔、决定何时停止、决定超时阈值。这在长等待过程中极易出现：忘记已轮询次数、提前终止、间隔不准确等问题。且整个轮询期间对话上下文持续占用资源。
- **根本原因**：`preAuditSubmit` 接口不允许非浏览器调用（沙箱 curl 返回 404），被迫通过 `agent-browser eval` 执行，导致轮询逻辑无法脚本化。
- **短期修复**：将 browser eval 的 JS 轮询代码固化为模板字符串，由脚本组装参数后注入，而非每次由 LLM 生成。
- **根本修复**：向后端提 issue，要求 `preAuditSubmit` 支持服务端直调；理想形态是后端支持 webhook 回调，Agent 从主动轮询改为被动唤醒。

---

**P1-3　轮询间隔在两处文档中不一致（10s vs 15s）**

- **位置**：`pre_review_submit_spec.md` 主体（10s）vs 附录（15s）
- **问题**：LLM 读到不同位置会使用不同间隔，导致行为不可预测，且无法被测试验证。
- **修复**：主体保留 10s（与 6 分钟超时匹配，36 次），附录标注"已废弃，以主体为准"。

---

**P1-4　全部 8 个文档一次性注入上下文，上下文过重（12-Factor F3 违反）**

- **问题**：初始注入约 5,900 token 的文档，加上对话历史，步骤⑦执行时上下文轻易超过 15k token。而步骤⑦是整个流程中最复杂的任务（解析 JSON + 生成风险报告），却在注意力最分散的时刻执行。
- **修复**：按步骤动态注入，每步只加载该步需要的 1-2 个文档：

  | 步骤 | 注入文档 |
  |------|---------|
  | ② 文件上传 | `file_upload_spec.md` |
  | ③ 模板识别 | `template_identify_spec.md` + `interaction_templates.md#步骤③` |
  | ④ 清单选择 | `checklist_query_spec.md` + `interaction_templates.md#步骤④` |
  | ⑥ 提交轮询 | `pre_review_submit_spec.md` |
  | ⑦ 结果输出 | `output_spec_pre_review.md` |

  单步上下文从 ~6k 降至 ~1-2k，格式遵守率更高，成本降低约 60%。

---

**P1-5　触发机制无豁免路径，"分享文件"会误触发预审**

- **位置**：SKILL.md 触发规范
- **问题**：触发规则是"收到文件链接即视为发起审查请求，禁止等待用户确认"。但用户可能只是在对话中分享合同文件给同事参考，或说"把上次的合同发给我看看"。这两种场景会触发完整预审流程，严重打断用户意图。
- **修复**：在触发后增加轻量意图判断——若用户消息同时包含"看看/发我/参考/下载/备份"等纯查阅意图词，询问"是否需要预审"再启动流程；否则保持立即触发。

---

**P1-6　`log_spec.md` 被引用但不存在于 zip 包和 manifest——悬空引用**

- **位置**：`file_upload_spec.md` 末尾"后续步骤"节
- **问题**：文档写道"日志打点（`log_spec.md`）的 `inputFileS3` 和 `inputFileName` 字段"，但该文件既不在 skill.manifest 的哈希清单里，也不在 zip 包中。LLM 被指示参考这个文件，但文件不存在，会导致步骤①/⑧的日志打点参数组装静默失败或出现幻觉字段。
- **修复**：补充创建 `references/common/log_spec.md` 并加入 manifest；或从 `file_upload_spec.md` 中删除对该文件的引用，将日志字段说明内联到步骤①/⑧的描述中。

---

### 🟡 P2 — 一致性问题（6 项）

---

**P2-1　输出消息条数在三处文档中给出三种答案**

- **位置**：SKILL.md 执行纪律（"合并一条"） vs `interaction_templates.md` 步骤⑦（"分两条"） vs `output_spec_pre_review.md`（合并，链接追加在末尾）
- **问题**：LLM 根据读到的文件内容决定发几条消息，行为随机漂移。用户在不同时间会看到不同格式的输出。
- **修复**：在 SKILL.md 唯一锚定为"合并一条消息"，其余两处文档删除矛盾表述，全部引用 SKILL.md 的规定。

---

**P2-2　output_spec_pre_review.md 正文禁止生成摘要，末尾却描述摘要生成策略**

- **位置**：`output_spec_pre_review.md` 第一段 vs 末尾"合同摘要生成策略"节
- **问题**：正文明确写"不要输出合同摘要段落，LLM 从文件名猜测极不可靠"；末尾却详细说明如何从文件名、模板信息、审查项生成 2-4 句摘要。两处来自不同时期的编辑，完全自相矛盾。
- **修复**：删除末尾的摘要生成策略（遵守正文的设计决策，接口未返回结构化摘要字段时不应由 LLM 猜测）。

---

**P2-3　interaction_templates.md 步骤③-A 存在重复的用户回复处理逻辑**

- **位置**：`interaction_templates.md` 情况 A（我方模板确认）
- **问题**：同一段落中出现了两套用户回复处理逻辑——一套有三个选项（正确/识别有误/对方模板），一套只有两个选项（是/不是）。后者是早期版本的残留，未被清除，LLM 会随机使用任意一套。
- **修复**：保留三选项版本（更完整），删除两选项的旧版本。

---

**P2-4　traceId 构造依赖毫秒时间戳，但未说明在 Agent 上下文中如何获取**

- **位置**：SKILL.md 步骤①
- **问题**：`traceId = {毫秒时间戳}_{mis}` 要求毫秒级时间戳，但 Claude Code workflow 环境中 `Date.now()` 不可用，文档未给出替代方案。LLM 会幻构一个假时间戳或直接失败。
- **修复**：补充明确的获取方式：`date +%s%3N`（Linux/macOS 均支持），并将这一步从"LLM 生成"改为脚本调用。

---

**P2-5　references/ 文件被当作静态附录，缺少步骤级导航指令**

- **位置**：SKILL.md 文件索引节
- **问题**：官方指南要求 references/ 文件是"LLM 按需导航发现"的知识节点（第三层渐进披露），而不是一次性全量加载的附录。当前的文件索引是静态表格，没有在各步骤处给出明确的导航指令，LLM 无法知道"执行步骤③时才需要读 template_identify_spec.md"。
- **修复**：在工作流各步骤说明中嵌入 `read` 指令，例如：
  ```
  ③ 模板识别（read references/common/template_identify_spec.md）
  ④ 清单选择（read references/common/checklist_query_spec.md）
  ```

---

**P2-6　缺少测试规范：无触发测试用例、无功能测试用例、无基线对比数据**

- **位置**：整个 Skill 包，无对应文件
- **问题**：官方指南要求 Skill 在分发前完成三类测试：① 触发测试（Should trigger / Should NOT trigger 列表）、② 功能测试（Given/When/Then 用例）、③ 有/无 Skill 的基线对比（token 消耗、API 调用次数）。contract-review 完全没有测试规范文件，导致每次迭代无法验证改动是否引入了触发漂移或功能退化。
- **修复**：新增 `tests/` 目录，包含 `trigger-tests.md` 和 `functional-tests.md`（至少覆盖我方模板、对方模板、识别失败三条核心路径）。

### 🟢 P3 — 优化项（6 项）

---

**P3-1　工具调用结果未标准化，LLM 直接解析深层嵌套 JSON**

- **问题**：模板识别结果（`data.data.resultDTOList[0].templateCode`）、预审状态（`data.data.auditBillStatus`）等均为三层嵌套结构，直接交给 LLM 解析。字段路径越深，LLM 出错概率越高。前面发现的 `auditItemResults` vs `results` 字段名错误正是由此引起。
- **建议**：脚本层将原始响应提取为扁平化结构后再传给 LLM：
  ```json
  {"found": true, "templateCode": "TP251121000014", "templateVersion": 1}
  ```

---

**P3-2　Prompt 分散在 8 个文件，无版本对齐机制（12-Factor F2 违反）**

- **问题**：SKILL.md 是 V21，但各 reference 文件没有独立版本号，不清楚哪个文件与 V21 对应。任何一处修改都可能在其他文件产生副作用，且无法追踪。
- **建议**：在每个 reference 文件头部增加 `last-updated-for-skill-version: V21` 标记；重大修改时建立 changelog。

---

**P3-3　additionalNotes 的 500 字截断依赖 LLM 自觉控制**

- **问题**：接口限制补充审查事项最多 500 字，但截断逻辑目前是 LLM 隐式处理。LLM 可能超出限制导致接口报错，或截断方式不合理（截到句子中间）。
- **建议**：脚本层在提交前执行 `text[:500]` 截断，并在截断时告知用户"内容已截取前 500 字"。

---

**P3-4　示例模板编号 TP240422000446 是否真实存在，用作示例有歧义**

- **位置**：`template_search_spec.md`（出现 3 次）
- **问题**：如果这个编号不存在于系统中，用户按格式输入后系统返回空，造成不必要困惑。
- **建议**：替换为真实可查询的模板编号，或明确加注"（仅为格式示例，非真实模板）"。

---

**P3-5　SKILL.md 指令优先级结构不合理，核心约束未置顶**

- **位置**：SKILL.md 整体结构
- **问题**：官方指南建议最关键的约束应排在最顶部（使用 `## CRITICAL` 等标题突出）。当前顺序是：触发规范 → 执行纪律 → 前置检查 → 工作流 → 文件索引。"执行纪律"（4 条强制规则）排在"前置：确保 CLI 最新"之后，优先级没有通过结构传递给 LLM。
- **建议**：重排为：⛔ 核心约束 → 🚀 触发规范 → 前置检查 → 工作流 → 文件索引。

---

**P3-6　description 触发短语以散文形式描述，不如枚举列表清晰**

- **位置**：SKILL.md frontmatter description 字段
- **问题**：官方指南要求 description 包含"用户实际会说的具体触发短语"枚举列表。contract-review 的触发条件以斜线分隔散文形式呈现，不如 contract-query-skill 的关键词列表对模型友好。
- **建议**：改为 `触发词：合同审查、合同预审、风险检查、审核合同、帮我看看这份合同、合同有没有风险` 枚举格式。

## 架构层建议：LLM 与脚本的边界重划

下图是当前实现（左）与建议实现（右）的对比：

```
【当前】                              【建议】
─────────────────────────────         ─────────────────────────────
LLM 驱动整个工作流                    脚本驱动工作流，LLM 只做翻译
  ├─ 状态管理（记在上下文里）    →      脚本维护 session.json
  ├─ 控制流跳转（LLM 判断）      →      代码状态机（match step: ...）
  ├─ 接口调用（LLM 组装执行）    →      脚本封装，LLM 只输出参数 JSON
  ├─ 轮询循环（LLM while）       →      脚本 while loop / webhook 回调
  ├─ 错误处理（LLM 临时决策）    →      脚本捕获 → 结构化错误事件 → LLM
  └─ 结果格式化（LLM）           →      脚本计数 + LLM 生成自然语言
```

**LLM 唯一不可替代的职责**：

| 步骤 | LLM 应做的事 |
|------|------------|
| ③ 模板来源确认 | 把用户回复（"就是这个"/"不对"/"对方的"）归一到 enum |
| ③-C 模板搜索 | 解析用户输入类型（TP编号 / 链接 / 关键词），提取搜索参数 |
| ④ 清单选择 | 把"1,3,5"/"第一个和第三个" 提取为序号数组 |
| ⑤ 补充事项 | 识别"跳过"意图 vs 收集自由文本 |
| ⑦ 风险报告 | 将 `riskDescript` + `thinkDesc` 转为人话描述和修改建议 |

**脚本应承担的所有其他工作**（占全流程约 70%）：SSO Cookie 构造、文件上传、任务轮询、状态持久化、序号映射、接口调用、数量统计、链接构造、消息发送。

## 五个评审框架的分工与互补

| 评审框架 | 核心关注点 | 发现的问题类型 | 代表性问题 |
|---------|-----------|--------------|-----------|
| **文档严谨性**（第一轮） | 文档内容的正确性与一致性 | 内容错误、矛盾、遗漏 | P0-1 billId串号、P0-2 字段名错误、P2-x 多处矛盾 |
| **LLM/脚本边界**（第二轮） | 每个步骤应由谁执行 | 职责错配、架构取向问题 | P0-3 序号映射、P1-1 控制流、P1-2 轮询 |
| **12-Factor Agents**（第三轮） | Agent 工程架构的正确性 | 结构性缺陷，看不出来的深层问题 | P0-4 无状态持久化、P1-4 上下文过重 |
| **官方 Skill 指南**（第四轮） | Skill 作为产品的完整性 | 产品缺口，影响可交付质量 | P1-6 悬空引用、P2-6 测试规范缺失 |
| **知识锚定**（第五轮） | LLM 的判断依据是什么 | 能力误用、知识真空 | P0-5 审查智能完全在黑箱、P1-7 LLM 无法补充判断 |

五个框架发现的问题**类型完全不同，没有重叠**——这正说明需要多框架交叉审查，任何单一视角都会遗漏重要问题。

## 知识锚定分析（第五轮）

**Skill 类型判断**：B 类——流程 + 专业判断混合型

contract-review 的表面是流程（收集参数→调接口→输出结果），但其核心价值主张是"合同审查"，这是一个需要法律知识支撑的专业判断任务。因此维度五完整适用。

### 当前知识来源的实际分工

```
用户上传合同文件
    ↓
【后端预审接口】← 真正的"审查智能"在这里，完全黑箱
  规则引擎扫描 → 返回结构化风险项列表
    ↓
【LLM】← 只做这件事：
  风险项 JSON → 格式化为 Markdown 摘要
    ↓
用户看到报告
```

**职责倒置的全貌**：

| 任务 | 擅长的执行者 | 实际执行者 | 错配程度 |
|------|------------|-----------|---------|
| 条款语义理解 | LLM | 后端黑箱规则引擎 | ❌ 严重错配 |
| 跨条款关联风险识别 | LLM | 无（规则引擎逐条扫描，无法跨条款） | ❌ 能力真空 |
| 法律依据引用 | LLM + RAG | 无 | ❌ 能力真空 |
| 修改建议生成 | LLM | LLM（搬运 thinkDesc 字段） | ⚠️ 无知识支撑 |
| 用户补充事项针对性检索 | LLM + RAG | 后端（透传，不一定有对应规则） | ❌ 严重错配 |
| JSON 格式化输出 | 任何模板引擎 | LLM | ❌ 大材小用 |

### 五个检查点评分

**5.1 判断来源透明度：1/3**
后端预审接口返回 `riskDescript`、`thinkDesc` 等字段，但这些字段的生成逻辑完全不透明——用户无法知道"为什么这个条款有风险"的法律依据是什么。Skill 输出的报告中没有任何法律条文引用，没有规则 ID，没有来源标注。LLM 以搬运接口结论的口吻呈现，用户无法区分"原文答复"和"推理建议"。扣 -2 分。

**5.2 知识库 / RAG 接入：0/3**
LLM 侧完全没有接入任何知识库。用户在步骤⑤填写的"补充审查事项"（如"重点关注知识产权归属"）只是透传给后端 `additionalNotes` 字段——LLM 无法基于法律知识对此做针对性检索和分析。后端规则引擎覆盖的是格式合规和已知违规条款，无法覆盖语义层的法律合规性判断。扣 -3 分。

**5.3 知识时效性：0/1**
后端规则库版本完全不透明，Skill 文档中无任何法律版本标注（如"依据 2021 年版《民法典》"）。合同法律环境随时可能变化，用户无法判断审查结论的时效性。扣 -1 分。

**5.4 兜底策略：0/2**
后端返回 0 个风险项时，Skill 直接输出"未发现风险"，没有任何 LLM 兜底检查。这意味着如果后端规则库没有覆盖某类风险（如某类新型合同条款），用户会收到错误的安全感。LLM 本可以对合同文本做一轮自主检查，但目前架构中完全没有这个环节。扣 -2 分。

**5.5 知识与操作分层：0.5/1**
流程上有分层——审查结果出来后用户可以选择继续提交或放弃，这是合理的。但 LLM 无法对结果做解释或追问（"这个条款的具体风险是什么？"），知识判断和操作执行之间没有真正的 LLM 介入窗口。扣 -0.5 分。

**维度五总分：1/10**

### 改进建议（按优先级）

**优先级 1：LLM 补充审查层（不依赖后端改造）**

在后端预审结果出来后，增加一个 LLM 主动审查步骤：
```
后端预审结果（风险项列表）
    ↓
LLM 读合同全文 + 结合用户补充事项
    → 对后端结论给出法律依据解释
    → 针对 additionalNotes 做专项检查
    → 识别后端未标注的语义层风险（标注"综合推理"）
    ↓
合并输出：后端规则命中 + LLM 语义补充
```

**优先级 2：RAG 接入法律知识库**

对 LLM 的补充审查接入外部知识库（如北大法宝、合同规范库），使 LLM 能够引用具体法条而非依赖训练数据记忆：
```
风险条款 → 向量检索相关法条 → LLM 生成带引用的风险说明
```

**优先级 3：后端规则引擎透明化（需后端配合）**

要求后端在风险项中返回规则 ID 和规则来源，LLM 可据此向用户展示"触发了哪条内部规则"，提升结论的可追溯性。

## 值得保留的优点

评审以问题为导向，但这些设计决策值得在改版中保留：

1. **故障显式标注**：CLI uploadFile 损坏、queryRules 不完整——直接写进文档并给替代方案，这是成熟工程文化的体现，应在其他 Skill 推广。
2. **降级策略一致**：模板识别失败→对方模板、清单接口失败→通用清单，所有失败路径都有明确的无损降级，不轻易打断用户。
3. **SSO 规范完整**：两个 clientId 分工、废弃方案列表、错误码处理、工程化约束表——这是文档体系中写得最严谨的文件。
4. **用户确认节点控制克制**：整个流程只有 3 个等待节点（模板确认/清单选择/补充事项），在流程效率和准确性之间取得了好的平衡，避免了过度打扰用户。
5. **超时兜底设计**：轮询超时后给出明确的落地链接（`/pre-review-list`），用户不会被卡死，这一点体验设计得周到。
